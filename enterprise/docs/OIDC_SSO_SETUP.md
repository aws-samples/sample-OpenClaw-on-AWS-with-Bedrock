# OIDC Single Sign-On Setup Guide

OpenClaw Enterprise supports any OpenID Connect (OIDC) compatible identity
provider, including:

- **Alibaba Cloud IDaaS** (EIAM)
- **Microsoft Entra ID** (Azure AD)
- **Okta**
- **Keycloak**
- Any other standards-compliant OIDC provider

All configuration is done **at runtime** through the admin console. No
environment variables or rebuilds required per tenant.

## How it works (BFF architecture)

```
Browser                     OpenClaw Backend              IdP
───────                     ─────────────────             ───

Click "Sign in with SSO"
  ↓
GET /api/v1/auth/sso/login
                            Generate state + PKCE verifier
                            Store in HttpOnly Cookie (10 min)
                            ← 302 to IdP /authorize ─────→
                                                          User authenticates
  ← 302 to
   /api/v1/auth/sso/callback?code=...
                            ← validate state from Cookie
                            POST /token with client_secret + verifier ──→
                                                          ← id_token
                            Verify id_token via JWKS (RS256)
                            Match email → DynamoDB employee
                            (or auto-provision if enabled)
                            Issue local HS256 JWT
                            ← 302 to /login/sso-success#token=xxx
  ↓
Frontend reads JWT from URL hash
Stores in localStorage
Redirects to /portal or /dashboard
```

**Key properties:**

- **Backend-for-Frontend (BFF)**: All OAuth interactions happen server-side.
  The browser never touches the IdP's token endpoint, so **no CORS config
  needed on the IdP side**.
- **Confidential Client** with `client_secret` + **PKCE** as defense in depth.
- **Local JWT** issued after successful IdP authentication — same token
  format as password login, so the auth middleware handles both uniformly.
- **Tokens in localStorage** (for the local JWT only, not IdP tokens).

## Prerequisites

1. An OIDC-compatible IdP with admin access
2. Your OpenClaw Portal must be reachable via **HTTPS** in production
   (HTTP is only allowed for `localhost`)
3. Your IdP application must support **Confidential Client** mode with a
   Client Secret (required — Public Client / PKCE-only mode is not supported)

## Step 1: Create an OIDC Application in Your IdP

### Common Settings (All Providers)

| Setting | Value |
|---|---|
| Application type | **Web Application** / **Confidential Client** (NOT SPA) |
| Grant type | **Authorization Code** (with PKCE supported) |
| Redirect URI | `https://your-portal.example.com/api/v1/auth/sso/callback` |
| Initiate Login URI (optional) | `https://your-portal.example.com/login?sso=idp` — enables IdP-initiated flow |
| Post Logout Redirect URI (optional) | `https://your-portal.example.com/login` |
| Scope | `openid profile email` |
| ID Token signing algorithm | RS256 |
| Client Secret required | **Yes** (generated by IdP console) |

> **Important**: The Redirect URI points to a **backend endpoint**
> (`/api/v1/auth/sso/callback`), not a frontend route. OpenClaw processes
> the OAuth callback entirely on the server.

### Alibaba Cloud IDaaS

1. Open your IDaaS instance console → **Applications** → **Add Application**
2. Choose **Custom OIDC Application**
3. Configure:
   - **Application Type**: OIDC SSO
   - **Client Type**: Confidential Client / 应用端机密客户端
   - **Authorization Mode**: `authorization_code`
   - **PKCE**: Enabled (S256)
   - **Redirect URI**: `https://your-portal.example.com/api/v1/auth/sso/callback`
   - **Initiate Login URI**: `https://your-portal.example.com/login?sso=idp`
4. After saving, copy:
   - **Issuer URL**, e.g.
     - Public entry: `https://eiam-api-{region}.aliyuncs.com/v2/{instance_id}/{app_id}/oidc`
     - Custom domain: `https://{prefix}.aliyunidaas.com/api/v2/iauths_system/oauth2`
   - **Client ID** (form `app_xxxxxxxxxx`)
   - **Client Secret** (shown once when app is created; regenerate in "Developer Access" / "Keys" section if lost)
5. Authorize the application to the employees who need access

### Microsoft Entra ID (Azure AD)

1. [Azure Portal](https://portal.azure.com) → **Microsoft Entra ID** → **App registrations**
2. **New registration**
   - **Supported account types**: *Accounts in this organizational directory only*
3. In the new app, go to **Authentication** → **Add a platform** → **Web**
   - **Redirect URI**: `https://your-portal.example.com/api/v1/auth/sso/callback`
   - **Front-channel logout URL** (optional): `https://your-portal.example.com/login`
   - Leave all implicit/hybrid flow checkboxes **unchecked**
4. **Important**: Do NOT add a "Single-page application" platform. Azure AD
   mandates that SPA-registered apps exchange tokens via the browser only,
   which is incompatible with BFF mode. If you see both SPA and Web platforms,
   delete the SPA platform.
5. **Certificates & secrets** → **New client secret**:
   - Description: e.g. `openclaw-sso`
   - Expires: 24 months (recommended)
   - Click **Add** and **immediately copy the Value** (shown only once)
6. Copy:
   - **Issuer**: `https://login.microsoftonline.com/{tenant_id}/v2.0`
     (tenant_id from Overview → Directory (tenant) ID)
   - **Client ID**: Overview → Application (client) ID
   - **Client Secret**: the value you just copied
7. API permissions → ensure `openid`, `profile`, `email` are granted

### Okta

1. **Applications** → **Create App Integration** → **OIDC - Web Application**
2. Configure:
   - **Grant types**: Authorization Code + Refresh Token (PKCE is enabled by default)
   - **Sign-in redirect URIs**: `https://your-portal.example.com/api/v1/auth/sso/callback`
   - **Sign-out redirect URIs**: `https://your-portal.example.com/login`
   - **Controlled access**: choose appropriate assignment policy
3. Copy:
   - **Issuer**: `https://{your-okta-domain}/oauth2/default`
   - **Client ID**
   - **Client Secret** (Client Credentials section)

### Keycloak

1. In your realm: **Clients** → **Create client**
2. Configure:
   - **Client type**: OpenID Connect
   - **Client authentication**: **ON** (confidential client)
   - **Authentication flow**: check only **Standard flow**
   - **Valid redirect URIs**: `https://your-portal.example.com/api/v1/auth/sso/callback`
   - **Valid post logout redirect URIs**: `https://your-portal.example.com/login`
   - **Advanced → Proof Key for Code Exchange Code Challenge Method**: S256
3. After creation, go to **Credentials** tab and copy the **Client secret**
4. Copy:
   - **Issuer**: `https://{keycloak-host}/realms/{realm-name}`
   - **Client ID**: the ID you set in step 1
   - **Client Secret**: from step 3

## Step 2: Configure SSO in OpenClaw

1. Sign in to the admin console with an **admin account** using password
2. Go to **Settings** → **SSO** tab
3. Note the `Redirect URI` and `Initiate Login URI` displayed — these must
   match what you registered in the IdP
4. Fill in:
   - **Issuer URL**: from your IdP console
   - **Client ID**: from your IdP console
   - **Client Secret**: from your IdP console
   - **Scopes**: `openid profile email` (default)
5. Click **Test Connection**
   - Backend fetches `{issuer}/.well-known/openid-configuration`
   - Success means the Issuer is reachable and returns valid OIDC metadata
6. Configure **Auto-Provisioning** (optional):
   - **Auto-create employees on first SSO login**: default on
   - **Default Position**: select from your position list
   - **Default Role**: `employee` (or manager/admin)
7. Check **Enable SSO** to show the "Sign in with SSO" button on login page
8. Click **Save**

The configuration takes effect immediately. No restart required.

## Step 3: Link Employees to SSO Identities

The backend identifies SSO users by the `email` claim in the ID token. There
are two ways to link SSO users to employees:

### Option A: Auto-Provisioning (default, recommended)

**Enabled by default**. When an SSO user signs in for the first time and
no matching employee exists, OpenClaw will automatically create:

- A new employee record (id based on email prefix, e.g.
  `jane.doe@company.com` → `emp-jane.doe`; collisions get a
  random 4-char suffix like `emp-jane.doe-a3f0`)
- A personal agent with skills inherited from the configured Default Position
- An audit log entry (`eventType: employee_auto_create`, `createdVia: sso_auto`)

After first login, the admin can refine the employee's position/role/skills
via **Organization → Employees** or **Agent Factory**.

### Option B: Manual Provisioning

Disable auto-create to require admins to create employee records manually
before anyone can sign in. Use this for tighter control.

**Organization → Employees** → add an employee → set the Email address
matching their IdP login.

Validation rules (applied both in UI and backend):

- Email is **optional** — employees who never sign in via SSO can leave it empty
- If provided, must be a valid email format
- Must be **unique** across the organization
- Stored in **lowercase** (case-insensitive matching)

When an SSO user signs in:

1. IdP issues an id_token with `email: jane.doe@company.com`
2. OpenClaw backend finds the employee with that email
3. User is signed in with the employee's role, department, agent, etc.

If no matching employee exists and auto-create is disabled, the user sees:
`SSO 登录成功,但未找到邮箱为 xxx 的员工。请联系管理员。`

## Login Flows Supported

### SP-Initiated (user starts at Portal)

```
User opens https://portal.example.com
  → Redirected to /login
  → Clicks "Sign in with SSO"
  → Redirected to IdP
  → Authenticates
  → Redirected back, signed in
```

### IdP-Initiated (user starts at IdP workspace)

Triggered automatically by a `?sso=idp` URL parameter. Configure the
**Initiate Login URI** in your IdP as `/login?sso=idp`.

```
User clicks OpenClaw icon in IdP workspace
  → IdP redirects to https://portal.example.com/login?sso=idp
  → Portal immediately triggers SSO login (no manual click)
  → IdP recognizes existing session, silently issues token
  → User lands in Portal with no manual steps
```

### Password (fallback for administrators)

Always available. Admin can sign in with employee ID + password even if SSO
is misconfigured or the IdP is unreachable.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Click "Sign in with SSO" → IdP shows "Invalid redirect_uri" | Redirect URI in IdP doesn't exactly match OpenClaw's | Ensure `https://your-portal.example.com/api/v1/auth/sso/callback` is registered in the IdP (exact host, port, protocol, path, no trailing slash) |
| Successful IdP login but error "未找到邮箱为 xxx 的员工" | No DynamoDB employee has that email | Either enable Auto-Provisioning, or add the email to the employee record in **Organization → Employees** |
| Error "token_exchange_failed" — Azure AD logs show `AADSTS9002327` (SPA client-type) | Azure AD app has an SPA platform configured | Delete the SPA platform in Azure Portal → Authentication. Keep only Web platform. |
| Error "token_exchange_failed" — Azure AD logs show `AADSTS700025` (Client is public) | Same as above — SPA platform makes app public | Same fix — delete SPA platform |
| Error "token_exchange_failed" — Azure AD logs show `AADSTS7000218` (client_secret required) | Client Secret missing or incorrect | Generate a new Client Secret in IdP console and paste into OpenClaw Settings → SSO |
| Test Connection fails with "HTTP error" | Backend cannot reach IdP (network/firewall) | Verify the EC2 instance has egress to the IdP; check security groups and NAT |
| Settings saved but login still uses old config | Caches on backend | Not an issue — backend clears caches automatically on save |
| Token expired unexpectedly | id_token lifetime shorter than expected | Users will be redirected back to /login; they can re-authenticate |

## Security Notes

- **Confidential Client with Client Secret** — standard OAuth 2.0 BFF pattern.
  The secret is stored in DynamoDB and only used server-side, never sent
  to the browser.
- **PKCE is used as defense in depth** on top of client_secret authentication,
  protecting against authorization code interception in transit.
- **Tokens stored in browser localStorage** (local HS256 JWT only) — cleared
  on logout. IdP-issued access/refresh tokens never leave the backend.
- **All token verification server-side** — the backend validates the RS256
  signature using the IdP's JWKS endpoint, plus `audience`, `issuer`, and
  `exp` claims.
- **HTTPS required in production** — only `localhost` is allowed for HTTP.
- **No refresh token management** — when the local JWT expires, the user is
  redirected back to /login. They can sign in again via SSO (silent if IdP
  session still alive) or password.
- **CSRF / state** — stored in HttpOnly Cookie with SameSite=Lax; validated
  strictly on callback.

## Disabling SSO

To turn off SSO while keeping the configuration:

1. Settings → SSO → uncheck **Enable SSO** → Save

To fully remove the configuration:

1. Settings → SSO → clear Issuer/Client ID/Secret → uncheck Enable SSO → Save

Users can still sign in with password while SSO is disabled.
